# Project Requirements — Team Collaboration Platform

You are building a multi-tenant SaaS product where teams can manage projects, assign tasks, chat in real-time, and collaborate — all within their own isolated workspace.

This document tells you **what to build and what rules to follow.**  
It does not tell you how to build it. That is your job.

---

## Read This First

Before writing a single line of code, you must be able to answer:

- What tables will you need?
- How are they related to each other?
- How will you make sure one team never sees another team's data?
- What breaks in your schema if the product grows 10x?

If you cannot answer these, you are not ready to build yet. Design first.

---

## The Product (Plain English)

Teams sign up and get their own workspace. Inside a workspace, they create projects. Inside projects, they create and assign tasks, chat with each other, and upload files. Admins manage who can do what. Teams can upgrade to a Pro plan to unlock more features.

---

## Non-Negotiable Constraints

These are rules. You may not break them.

1. **A user can belong to multiple workspaces.** Each workspace is completely isolated. No user should ever be able to access, see, or modify data belonging to a workspace they are not a member of — not through the UI, not through the API, not by guessing an ID.

2. **Roles exist at the workspace level.** Not every member can do everything. You must enforce who can do what. This must be enforced on the server — never trust the frontend.

3. **Passwords are never stored in plain text.** Ever.

4. **Authentication must use short-lived access tokens and long-lived refresh tokens.** Understand why before you implement it.

5. **Every API route that touches workspace data must verify the requesting user actually belongs to that workspace.** This check cannot be optional.

6. **File metadata must be stored in the database. Files themselves must not be stored on your server.**

7. **Emails and reminders must not block the main API response.** If sending an email takes 3 seconds, the user should not wait 3 seconds for their API call to return.

8. **Pro plan limits must be enforced on the server.** A free workspace cannot exceed its limits by calling the API directly.

9. **The database schema must be decided and documented before you write any backend code.** No changing core tables halfway through.

10. **Real-time features must work across multiple users simultaneously.** Test with two browser tabs minimum.

## Features

---

### 1. Authentication

**What users can do:**
- Register with email and password
- Log in with email and password
- Log in with their Google account
- Stay logged in across browser refreshes
- Log out from their current session
- Log out from all sessions

**Before you build, answer these:**
- How will you handle Google login for a user who already registered with email? Same account or two accounts?
- Where do you store the refresh token — in the database, in a cookie, in localStorage? What are the tradeoffs of each?
- What happens if someone steals a user's access token? How does your system limit the damage?
- How do you invalidate all sessions when a user logs out of everything?
- What information goes inside the JWT payload? What should never go inside it?

**Acceptance criteria:**
- A user can register, log in, and access protected routes
- An expired access token returns 401
- A valid refresh token issues a new access token
- Google OAuth creates or links an account correctly
- Logging out invalidates the session

---

### 2. Workspaces

**What users can do:**
- Create a workspace with a name
- Invite other users to their workspace by email
- Assign roles to members (owner, admin, member)
- Remove members from a workspace
- Transfer ownership to another member
- Delete a workspace (only the owner)

**Before you build, answer these:**
- A user is a member of three workspaces. How does your schema represent this without duplicating the user?
- What happens to all the projects and tasks if a workspace is deleted?
- If a user is removed from a workspace, but they are the assignee on 12 tasks — what happens to those tasks?
- How do you make sure a workspace slug or name is unique, but only within the platform, not globally per user?
- What is the minimum role required to invite someone? How do you enforce that?

**Acceptance criteria:**
- A user can create a workspace and becomes its owner automatically
- Inviting a user sends them an email and adds them on acceptance
- Only an owner can delete the workspace
- Roles are enforced — a member cannot invite or remove others
- Removing a user does not break existing task assignments

---

### 3. Projects

**What users can do:**
- Create a project inside a workspace
- View all projects they have access to inside a workspace
- Archive a project (it disappears from the main view but is not deleted)
- Delete a project
- Add specific workspace members to a project

**Before you build, answer these:**
- Can every workspace member see every project? Or do projects have their own membership? Design for both and pick one — justify your choice.
- How do you make sure a project always belongs to exactly one workspace and cannot be moved?
- If a project is deleted, what happens to all its tasks, chat messages, and files?
- How do you query "all projects this user has access to inside this workspace" efficiently?

**Acceptance criteria:**
- A project is always scoped to a workspace — it cannot exist without one
- Archived projects do not appear in the main list but can be viewed separately
- Deleting a project deletes all its child data
- A user outside the workspace cannot access the project even with a direct ID

---

### 4. Tasks

**What users can do:**
- Create a task inside a project with a title, description, assignee, due date, and priority
- Move a task through statuses (e.g. To Do → In Progress → Done)
- Assign or reassign a task to any project member
- Add a comment on a task
- Create a subtask under a task
- Reorder tasks within a status column (drag and drop order must persist)
- Attach files to a task

**Before you build, answer these:**
- How do you store the order of tasks within a column so reordering can be saved without updating every row?
- How do you model subtasks? A separate table? A self-referencing foreign key? What are the tradeoffs?
- A task has an assignee. The assignee leaves the workspace. What happens to the task?
- How do you query all tasks assigned to a specific user across all their workspaces in one call?
- Comments are ordered by time. Files are attached by users. Do these need separate tables? Why or why not?

**Acceptance criteria:**
- Task status, priority, assignee, and due date can all be updated independently
- Subtasks are linked to their parent task
- Reordering tasks persists after page refresh
- Commenting on a task notifies the assignee
- A user not in the project cannot create or view tasks

---

### 5. Real-Time Chat

**What users can do:**
- Send messages in a project's chat channel
- See messages from other users appear instantly without refreshing
- See when other members are online in the project
- Send a file in chat

**Before you build, answer these:**
- How does the server know which project room to broadcast a message to?
- How do you make sure a user can only connect to a WebSocket room for a project they are a member of?
- Chat history needs to load when a user opens the chat. How do you paginate old messages efficiently?
- If two users send a message at the exact same millisecond, how does your system handle ordering?
- What happens to WebSocket connections when the server restarts?

**Acceptance criteria:**
- Messages appear in real-time for all connected users in the project
- A user cannot connect to a project room they don't belong to
- Existing chat history loads on open (paginated)
- Presence (online/offline) updates in real-time

---

### 6. Notifications

**What users can do:**
- Receive an in-app notification when a task is assigned to them
- Receive an in-app notification when someone comments on their task
- Receive an email when they are invited to a workspace
- Receive an email reminder before a task is due
- Mark notifications as read

**Before you build, answer these:**
- Sending an email takes time. How do you make sure this does not slow down the API response that triggered it?
- How do you schedule a reminder email to go out 24 hours before a task due date — even if the server restarts in between?
- A task due date is changed after the reminder was already scheduled. How do you handle the old scheduled job?
- How do you store notifications in the database so you can efficiently query "all unread notifications for this user"?

**Acceptance criteria:**
- Assigning a task triggers a notification to the assignee without slowing the API
- Email reminders fire at the right time even after a server restart
- Changing a due date cancels and reschedules the reminder correctly
- Notifications can be marked as read individually or all at once

---

### 7. File Attachments

**What users can do:**
- Upload a file and attach it to a task
- View and download files attached to a task
- Delete a file they uploaded

**Before you build, answer these:**
- Where do the actual files live? Where does the metadata live? These are two different answers.
- How do you make sure only workspace members can download a file — not anyone with the URL?
- A task is deleted. How do you make sure the file in storage is also cleaned up and not left behind forever?
- What file size limit will you enforce, and where — frontend, backend, or storage level?

**Acceptance criteria:**
- Files are not stored on the application server
- File metadata (name, size, type, uploader, task) is stored in the database
- Downloading a file requires authentication and workspace membership
- Deleting a task cleans up its associated files

---

### 8. Pro Plan & Billing

**Free plan limits:**
- Max 3 projects per workspace
- Max 10 members per workspace
- No file attachments

**Pro plan:**
- Unlimited projects
- Unlimited members
- File attachments enabled
- Priority support

**What users can do:**
- Upgrade their workspace to Pro
- Manage their subscription (cancel, update payment)
- See their current plan on the settings page

**Before you build, answer these:**
- Where do you store whether a workspace is on the free or pro plan? How do you keep it in sync with the payment provider?
- The payment provider sends you an event when a subscription is cancelled. How does your server receive and process this?
- How do you enforce the 3-project limit on the free plan without checking it on every single request?
- A workspace downgrades from Pro to Free. They have 8 projects. What happens?

**Acceptance criteria:**
- A free workspace cannot create a 4th project — the API rejects it
- A free workspace cannot upload file attachments — the API rejects it
- Upgrading to Pro immediately removes the limits
- Cancelling a subscription eventually returns the workspace to Free (not immediately — handle grace period)
- Billing events from the payment provider update the workspace plan automatically

---

### 9. Search

**What users can do:**
- Search for tasks by title or description within their workspace
- Search for members by name or email within their workspace

**Before you build, answer these:**
- How do you make sure search results only return data the user is allowed to see?
- Full-text search on every query can be slow. How does your database support this efficiently?
- Should search be a separate service or part of the main API? What are the tradeoffs?

**Acceptance criteria:**
- Searching returns relevant tasks and members
- Results never include data from other workspaces
- Search works on partial words (e.g. "log" returns "login bug")

---

### 10. Analytics

**What workspace admins can see:**
- Total tasks created vs completed over time
- Tasks per member (who has the most open tasks)
- Overdue tasks count
- Project activity over the last 30 days

**Before you build, answer these:**
- Do you calculate analytics on the fly from raw data, or do you store pre-computed results? What are the tradeoffs?
- Analytics queries can be expensive. How do you prevent them from slowing down the rest of the API?
- How do you define "overdue" at the database level?

**Acceptance criteria:**
- Analytics are only visible to admins and owners
- Data is scoped strictly to the workspace
- Dashboard loads in under 2 seconds (use caching if needed)

---

## Definition of Done

A feature is not done when it works in your browser.  
A feature is done when:

- [ ] It works correctly for multiple users simultaneously
- [ ] It rejects requests from unauthorized users
- [ ] It handles missing or invalid input without crashing
- [ ] It does not expose data from other workspaces
- [ ] It works after a server restart

## Final Note

The hardest part of this project is not writing the code.  
It is designing the data model correctly before you write any code.

A wrong schema decision in week one will cost you three weeks of rewrites later.

Draw your tables. Draw the relationships. Ask yourself what breaks.  
Then build.
